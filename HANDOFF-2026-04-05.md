## 3. 这次已明确定位的 bug / 问题

### 3.1 第一个明确 bug：screen 层叠遮挡

#### 现象
用户看到按钮，但点击像没反应。

#### 根因
CSS 写法错误，导致多个 `.screen` 同时存在于页面布局中，虽然视觉上不一定都明显，但会互相遮挡、抢点击。

#### 错误思路（旧）
`.screen` 只是透明：

```css
.screen {
  opacity: 0;
}
```

这会导致元素仍然在布局里。

#### 正确思路（新）
默认隐藏，只有激活页面显示：

```css
.screen {
  display: none;
}

.screen.active {
  display: flex;
}
```

#### 这次修复状态
- 本地修复版：已验证通过
- 线上仓库 `styles.css`：已经通过 GitHub 网页端提交过修复

---

### 3.2 第二个明确 bug：线上 `script.js` 是残缺版

#### 现象
虽然 CSS 修了一部分，但线上页面仍然出现：
- `Let's Go!`
- `Next`
- `Find it!`
- `Play Again!`
同时暴露

切屏逻辑没有按预期工作。

#### 根因
线上仓库里的 `script.js` 不是完整文件，而是残缺版，只剩后面的一小截，缺少前面核心逻辑。

#### 影响
- `showScreen()` / 初始化 / 事件绑定等主逻辑不完整
- 页面结构在，行为不对
- 看起来像“页面有东西，但点不动 / 跳不对”

#### 当前状态
- 本地 `script.js` 是完整可运行版
- 线上 `script.js` 仍未成功整份覆盖
- GitHub 网页编辑器尝试提交过，但没有可靠写完整

## 4. 这次踩过的坑（重点复盘）

### 坑 1：不要把“缓存”当成默认答案

一开始容易怀疑：
- 浏览器缓存
- Pages 缓存
- 用户不会点

但这次真实根因不是缓存，而是：
- CSS 结构 bug
- 线上脚本文件残缺

**结论：先查 DOM / CSS / JS 文件完整性，再谈缓存。**

---

### 坑 2：GitHub 网页编辑器不适合可靠覆盖长脚本

对短文件 / 小修改：
- 还凑合能用

对长文件（像完整 `script.js`）：
- 很容易只改到当前可控区域
- 提交后看似成功，实则不是整份替换
- github.dev / 普通编辑页都不够稳

**结论：长文件不要依赖 GitHub 网页编辑器做整文件覆盖。**

---

### 坑 3：本地仓库能 commit，不代表能 push

这次本地子仓库：
- 可以 commit
- 不能 push

失败原因：
- remote 是 HTTPS：`https://github.com/yinshengkui/fluffy-memory.git`
- 当前环境没有：
  - `gh`
  - GitHub token
  - SSH push 权限
  - Keychain 中的 github.com 凭证

**结论：本地发版能力要单独验证，不要默认“能 commit 就能发”。**

---

### 坑 4：GitHub token 创建会触发 sudo mode

即便浏览器已经登录 GitHub，去创建 token 仍然会要求：
- `Confirm access`
- 输入 GitHub 密码

这一步必须由账号拥有者本人完成，不能假设浏览器已登录就能一路自动创建 token。

---

### 坑 5：不要只看仓库文件变了，要看线上页面行为

这次就出现过：
- GitHub 页面有新 commit
- 但线上行为仍然不对

可能原因：
- 文件没完整写进去
- raw/CDN 还没刷新
- 提交的是局部，不是完整文件

**结论：最终验收标准是页面行为，不是“看起来提交成功了”。**

## 5. 当前最推荐的继续推进方式

### 方案 A：优先把 GitHub push 能力真正打通（最推荐）

#### 原因
一旦 push 通了：
- 本地完整文件就能直接推
- 不需要再赌 GitHub 网页编辑器
- 后续所有修复效率都会大幅提升

#### 建议路径
1. 在 GitHub 页面完成一次 `Confirm access`
2. 创建 fine-grained PAT
3. 权限只给目标仓库 `fluffy-memory`
4. 内容权限至少给：`Contents: Read and write`
5. 本机配置 git HTTPS 凭证
6. 再执行：

```bash
cd ~/.openclaw/workspace/tmp/peppa-alpha-game
git push origin main
```

#### 配通后第一件事
直接把本地完整 `script.js` 推上去。

---

### 方案 B：改用 SSH push

如果本机已有可用 GitHub SSH key，也可以把 remote 切成 SSH：

```bash
git remote set-url origin git@github.com:yinshengkui/fluffy-memory.git
```

然后：

```bash
git push origin main
```

但这条路当前未验证，因为这台机器目前 `ssh -T git@github.com` 不通。

---

### 方案 C：继续网页端编辑，但只适合小改动

适合：
- 改少量 CSS
- 改 1~2 行 HTML
- 改很小的 JS 片段

不适合：
- 覆盖完整 `script.js`
- 大规模重写
- 复杂结构调整

## 8. 可扩展的外网可访问方案（不要只盯 GitHub Pages）

这次的项目是纯静态站点，所以外网部署选择很多，不需要把视野只锁死在 GitHub Pages。

### 8.1 GitHub Pages

#### 优点
- 零成本
- 对公开仓库很方便
- URL 固定
- 对简单静态页面足够

#### 缺点
- 发版路径依赖 GitHub 仓库写权限
- 如果本地 push 不通，就会很痛苦
- 网页端编辑长文件很差
- 调试体验一般

适合：
- 稳定展示页
- demo 页面
- 低频更新页面

---

### 8.2 Vercel

#### 优点
- 部署快
- 预览链接方便
- 对前端项目支持极好
- 绑定 GitHub 后自动发版
- 支持自定义域名和多环境

#### 缺点
- 仍然依赖 GitHub 授权
- 免费版有一些限制

适合：
- 高频迭代前端 demo
- 想要 preview URL
- 想以后扩到 React / Next.js

#### 适合这个项目的原因
这个小游戏即使以后从纯 HTML/CSS/JS 升级为 React 版本，Vercel 也很好接。

---

### 8.3 Netlify

#### 优点
- 对静态站点非常友好
- 拖拽部署也可以
- Git 挂钩发版简单
- 支持 forms / redirects / headers

#### 缺点
- 某些体验和生态整合不如 Vercel 流畅

适合：
- 纯静态页面
- 需要简单 CDN 托管
- 想要除了 GitHub Pages 外更灵活的静态站方案

---

### 8.4 Cloudflare Pages

#### 优点
- 全球 CDN 很强
- 速度通常不错
- 对静态站友好
- 后续也方便接 Workers

#### 缺点
- 初期配置对不熟的人稍微绕一点

适合：
- 面向公网访问速度要求高
- 后续想接轻量边缘逻辑

---

### 8.5 独立服务器 / VPS 静态托管

例如：
- Nginx 托管静态目录
- Docker + Caddy / Nginx
- 挂在已有云主机上

#### 优点
- 完全可控
- 不依赖第三方前端平台策略
- 后续可逐步加 API / 埋点 / 后台

#### 缺点
- 维护成本更高
- 需要自己管 TLS / 域名 / 运维

适合：
- 后续要把小游戏平台化
- 要积累多个小游戏
- 想统一放到一个域名或服务器体系里

---

### 8.6 局域网/临时穿透方案（适合快速演示，不适合正式长期）

如果只是临时外网访问本地服务，可考虑：
- ngrok
- Cloudflare Tunnel
- Tailscale Funnel
- frp（自建）

#### 优点
- 立刻把本地页面暴露到外网
- 非常适合临时演示和联调

#### 缺点
- 不适合作为正式长期托管主路
- 地址可能变化
- 依赖本地机器在线

适合：
- 先给别人看 demo
- 快速临时分享
- 不想先走正式部署流程

## 9. 后续扩展建议（保留视野，不锁死当前方案）

### 扩展方向 1：小游戏模板化
把当前项目抽成一个小游戏模版：
- 通用屏幕切换框架
- 通用进度条
- 通用奖励动画
- 通用题目配置

以后只改：
- 字母 / 单词 / 图片
- 主题色
- 背景角色

这样后续可以快速复制出：
- 字母 A-Z 系列
- 数字学习版
- 动物认知版
- 颜色识别版

---

### 扩展方向 2：数据配置化
把游戏内容从 `script.js` 里抽出来，改成 JSON 配置，例如：

- rounds.json
- theme.json
- assets.json

优点：
- 改内容不改逻辑
- 更适合批量生成多个小游戏
- 更适合以后接 CMS 或 AI 生成内容

---

### 扩展方向 3：平台化托管
如果后续不是只做一个小游戏，而是做一批，可以考虑：
- 一个统一首页
- 多个小游戏子路由
- 公共组件复用
- 公共部署流水线

比如：
- `/games/letters-a-c`
- `/games/numbers-1-10`
- `/games/animals-basic`

这时 GitHub Pages 可能就不是最优解，Vercel / Netlify / 自托管会更合适。

---

### 扩展方向 4：加入轻量后端能力
如果以后想做：
- 分数记录
- 游玩统计
- 用户分享
- AI 生成关卡

那就可以进一步考虑：
- Cloudflare Workers
- Firebase Hosting + Functions
- Supabase
- 小型 Node 服务

当前项目先不需要，但架构上值得留白。
